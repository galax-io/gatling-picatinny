# Implementation Plan: Template Output Fixes (v1.19.0)

**Branch**: `004-template-output-fixes` | **Date**: 2026-06-22 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `/specs/004-template-output-fixes/spec.md`

## Summary

Milestone 3 bundles five output-correctness fixes in the templates DSL, all confined to two
files — `templates/Syntax.scala` and `templates/Templates.scala`:

1. **EL detection in `arr()` (#74)** — recognize dotted/hyphenated EL names (`#{user.id}`, `#{tenant-name}`) when classifying array string elements.
2. **`arr()` truncation (#203)** — a string containing `#{...}` mixed with literal text (`"hello #{name}!"`) must render in full, not collapse to the extracted variable name.
3. **XML element-name escaping (#203)** — element names emitted into XML must be escaped the same way JSON already escapes them, closing a structural-markup-injection gap.
4. **`RawValGen` string escaping (#203)** — `RawValGen` values that are stringy must be escaped (and JSON-quoted) on emit; only genuine scalars (numbers, booleans) stay raw.
5. **Missing templates directory fast-fail (#73)** — dereferencing the template registry when the `templates` resource is absent from the classpath must raise an explicit, diagnostic error instead of silently producing an empty map that later fails deep in the scenario with `NoSuchElementException`.

Technical approach: pure string-rendering changes plus one regex widening and one
null-resource branch. No new dependencies, no public method-signature changes. The
Java facade (`TemplateSyntax`) delegates to the same core methods (`makeJson`, `makeXml`,
`arr`) and inherits every fix unchanged. All five changes alter generated request bytes
and are therefore a documented behavior change (see Constitution Check / FR-006).

## Test Model *(mandatory — real cases + test sketches, NO implementation)*

| Req | Real case to test | Layer | Test sketch (no code) |
|-----|-------------------|-------|-----------------------|
| FR-001 | `arr("#{user.id}")` and `arr("#{tenant-name}")` — dotted/hyphenated EL names | Unit / Functional | Render the array and assert the element is the full literal EL string (`"#{user.id}"`, `"#{tenant-name}"`), classified as EL-bearing. Negative/regression: `arr("plain")` renders as plain `"plain"` and is NOT classified as EL. |
| FR-002 | `arr("hello #{name}!")` — literal text surrounding an EL expression | Unit / Functional | Render and assert the whole string survives as one element (`["hello #{name}!"]`), no characters dropped. Boundary: `arr("#{a} and #{b}")` (two expressions) renders the full string intact. Regression: pure `arr("#{name}")` still renders `["#{name}"]`. |
| FR-003 | An XML field whose name contains `<` / `&` (e.g. element name `a<b`) and a value with special chars | Unit / Functional | `makeXml` escapes the element name so no new markup can be injected; assert exact escaped bytes. Regression (exact value): existing `makeXml("name" - "foo")` still yields `<name>foo</name>` — plain identifiers stay byte-identical to today. |
| FR-004 | `RawValGen("a\"b")` / `RawValGen("<x>")` (stringy) vs `RawValGen(42)`, `RawValGen(true)` (scalar) | Unit / Functional | In JSON a stringy `RawValGen` is quoted and escaped (`"a\"b"`); in XML it is escaped (`&lt;x&gt;`). Negative/exact-value regression: `RawValGen(42)` stays raw `42`, `RawValGen(true)` stays raw `true`, `RawValGen(3.14)` stays `3.14` (locks existing `SyntaxSpec` expectations). |
| FR-005 | Template registry dereferenced when the `templates` resource is absent from the classpath | Unit / Functional | With the thread context classloader pointed at a location that has no `templates` resource, force the registry's first access and assert an explicit error is raised whose message names the missing `templates` directory. Positive regression: with `src/test/resources/templates` present, the existing names (`test_json`, `test_xml`) still load. Boundary: a `templates` resource that exists but contains no files yields an empty registry (NOT an error) — distinct from missing. |
| FR-001/002/004 (facade) | `TemplateSyntax.makeXml(...)`, `TemplateSyntax.makeJson(...)`, `TemplateSyntax.fieldArr(...)` over the same inputs | Facade Delegation (JUnit 5) | Assert the Java facade output is byte-identical to the Scala core output for an escaped-value case (no facade-only logic; delegation only). Confirms Principle I after the rendering change. |
| FR-006 | Generated request bytes differ from v1.18.x | Release gate (not a code test layer) | Verified at release time, not by an automated test: ships as a conventional `fix!:`/breaking-flagged commit so git-cliff renders the ⚠️ behavior-change note in the GitHub Release. See Complexity Tracking — this FR is a documentation/process requirement with no runtime behavior to assert. |

**Layer rationale (TESTING.md — apply only what fits):**
- **Unit / Functional is authoritative here.** All four behavioral fixes are pure string rendering; exact-byte `shouldBe` assertions are stronger and more precise than any higher layer.
- **Full Gatling e2e (examples overlay) deliberately NOT added.** Whether Gatling actually interpolates `#{user.id}` at runtime is Gatling's EL engine's responsibility, not picatinny's — picatinny only emits the string. An e2e would test Gatling, not this change. (Recorded in research.md.)
- **External integration (Testcontainers) N/A** — no Redis/Vault/JDBC involved.
- **Compile guard not required** — no public signatures change.
- **Facade delegation required** — `TemplateSyntax` exists and is affected; one delegation test confirms parity.
